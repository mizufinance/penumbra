# Compliance Roadmap

## 1. Penumbra Integration

Ensure compliance doesn't break existing functionality. All tests must pass.

| Item | Description |
|------|-------------|
| Block regulated assets in other actions | Reject Swap, Delegate, IBC, Position, CommunityPool, Governance for regulated assets |
| Multi-spend transactions | Handle N spends → M outputs with proper leaf binding |
| Cross-action leaf binding | Transaction-level spend↔output validation |
| Integration tests | Re-enable all disabled tests, add compliance coverage |
| Error handling | Replace panics with Result types in compliance crate |
| Genesis asset config | Auto-register native assets at genesis |

## 2. Orbis / Registries

Key management, signatures, registries.

| Item | Description |
|------|-------------|
| Key management handoff | Orbis manages issuer MCK, derives user keys |
| Signature verification | Orbis signs asset/user registrations |
| Registry revamp | Analyze best implementation of registries |
| Historical anchor support | Accept past tree roots (like SCT) or implement retry |
| Asset tree mutability | Analyze: can assets be unregistered? Affects anchor validation |
| User onboarding API | Orbis-driven registration flow |
| Per-address registration | Each address key must be individually generated by Orbis and registered in Penumbra |

## 3. Security

Cryptographic security and key isolation.

| Item | Description |
|------|-------------|
| BLACK_HOLE_ACK audit | Verify usage is secure |
| Ciphertext Randomness | Verify CSPRNG, no ephemeral reuse |
| Constant-time encryption | Same timing for regulated/unregulated paths |
| Detection key isolation | Dual master key architecture (separate detection from data trees) |
| ZK circuit verification | Formal verification of extended spend/output circuits |
| Daily key evaluation | Evaluate if we keep daily master key, or just the 3 daily key types is enough |

## 4. Client UX

Improve client experience and performance.

| Item | Description |
|------|-------------|
| Local compliance trees | Sync trees locally like SCT (no network queries at tx time) |
| Auto-registration | Register user on first regulated transfer |
| Wallet SDK hooks | APIs for wallet integration |
| Clear error messages | Descriptive errors when compliance fails |
| Event indexing | pindexer support for compliance events |

## 5. Benchmarks

| Item | Description |
|------|-------------|
| Validator proof verification | Time to verify extended spend/output proofs |
| Validator anchor checks | Registry lookup overhead |
| Client proof generation | Time for compliance-extended proofs |
| Client state sync | Compliance tree sync overhead |
| Scanner throughput | Transactions scanned per second |
| Decryption performance | Time to decrypt compliance ciphertext |
| Orbis interaction latency | Key derivation and API round-trips |

## 6. Future Features

| Item | Description |
|------|-------------|
| Threshold/whitelist | Per-asset transfer limits and flag, KYC whitelists |
| Partial address ciphertexts | Analysis full address vs partial address |
| Key rotation protocol | Evaluate future support |
| Swap support | Analysis of swap and trade support |
| Asset type support | Registry must handle all asset types (RWA, NFT, ERC20, etc.) |

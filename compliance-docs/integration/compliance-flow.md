# Compliance Flow

End-to-end overview of the compliance process from registration to auditing.

## 1. Asset Registration (Issuer)

Issuer registers asset as regulated on-chain.

```
Issuer → RegisterAsset { asset_id: USDC, is_regulated: true }
         → Asset added to asset tree
         → Issuer receives AMK (Asset Master Key)
```

## 2. User Registration (Alice & Bob)

Each user registers their address for the regulated asset.

```
Alice → RegisterUser { address: alice_addr, ACK: alice_ack, asset_id: USDC }
        → Alice's compliance leaf added to user tree

Bob   → RegisterUser { address: bob_addr, ACK: bob_ack, asset_id: USDC }
        → Bob's compliance leaf added to user tree
```

**Note:** Each address must be individually registered. ACK is generated by Orbis.

## 3. Pre-Transaction (Bob shares data with Alice)

Before Alice can send to Bob, she needs Bob's compliance leaf for encryption.

```
Bob   → shares compliance leaf (address + ACK) with Alice
        (out-of-band or queried from registry)
```

## 4. Transaction (Alice sends to Bob)

Alice builds and sends the transaction.

```
Alice → Build tx: Spend(alice_note) + Output(bob_addr, amount)
        → Planner enriches with compliance data:
           - Fetches Merkle proofs from registry
           - Generates sender ciphertext (for alice's ACK)
           - Generates receiver ciphertext (for bob's ACK)
        → Proofs generated binding compliance data
        → Transaction broadcast
        → Validator verifies proofs + anchors
        → Ciphertexts stored on-chain
```

## 5. Scanning
In production Orbis will share only the detection key with the issuer. The issuer would send the tx of interest to Orbis for re encryption.

For the demo, we bypass Orbis and share all daily keys with the scanning module, which detect transactions and then decrypt them.
